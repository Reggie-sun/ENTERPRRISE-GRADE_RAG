# DFMS嵌入式智能助手架构方案（领导汇报版 V2）

## 1. 项目结论

本项目不是单独做一个 AI 工具，而是在 DFMS 之上建设一个可长期演进的智能助手体系。

这个体系的核心作用，是把原本分散在系统界面、技术资料、报警处理、生产数据中的信息能力整合起来，形成一个统一的智能入口，服务三类核心人群：

- 一线操作人员
- 设备运维与工程师
- 生产管理与经营管理人员

从管理层视角看，这个项目最终要形成的不是“一个问答功能”，而是一个可以支撑产品升级、客户交付和商业化推广的智能化架构。

---

## 2. 这套系统最终会长成什么样

项目落地后，DFMS 智能助手会以 3 个产品形态存在：

### 2.1 DFMS 内嵌助手

直接嵌入 DFMS 页面，用户在设备监控、报警、工单、报表等页面中可随时调用。

主要作用：

- 页面解释
- 操作引导
- 知识问答
- 故障辅助
- 数据分析

### 2.2 企业微信助手入口

作为移动端和通知入口，用于承接：

- 报警推送
- 工单提醒
- 维修建议推送
- 智能助手快捷进入

### 2.3 管理运营后台

面向内部运营和交付团队，负责：

- 文档上传和知识更新
- 知识审核
- 权限范围配置
- 应用运营和使用分析

## 3. 这套架构到底解决谁的问题

| 角色 | 当前问题 | 使用入口 | 系统提供的核心能力 |
|---|---|---|---|
| 新员工 / 操作员 | 系统复杂、不会用、培训周期长 | DFMS 页面 | 页面解释、操作引导、知识问答 |
| 运维工程师 | 报警后资料难找、处理慢、经验不易沉淀 | DFMS + 企业微信 | 报警联动、维修建议、知识引用 |
| 生产管理人员 | 数据难查、分析依赖 IT | DFMS + 企业微信 | 自然语言查数、图表分析、结论输出 |
| 管理员 / 运营人员 | 文档分散、权限难管、效果难追踪 | 管理后台 | 知识维护、审核、权限、监控、审计 |

这张表的意思很明确：

本项目不是只服务某一个岗位，而是围绕 DFMS 形成一个覆盖“使用、运维、管理”的统一智能架构。

## 4. 总体业务架构

整体架构建议按 5 层来看。

```text
┌────────────────────────────────────────────────────────────┐
│                       第一层：使用入口层                    │
│ DFMS 页面 / 企业微信 / 管理后台 / 外部业务系统              │
└──────────────────────────┬─────────────────────────────────┘
                           │
                           ▼
┌────────────────────────────────────────────────────────────┐
│                       第二层：应用服务层                    │
│ 智能问答服务 / 报警辅助服务 / 数据分析服务 / 知识运营服务   │
└──────────────────────────┬─────────────────────────────────┘
                           │
                           ▼
┌────────────────────────────────────────────────────────────┐
│                       第三层：智能中台层                    │
│ 上下文识别中心 / 知识服务中心 / 数据分析中心 / 消息触达中心 │
│ 权限审计中心                                               │
└──────────────────────────┬─────────────────────────────────┘
                           │
                           ▼
┌────────────────────────────────────────────────────────────┐
│                      第四层：业务与数据层                   │
│ DFMS 业务数据 / 技术文档 / 报警数据 / 工单数据 / 用户组织   │
└──────────────────────────┬─────────────────────────────────┘
                           │
                           ▼
┌────────────────────────────────────────────────────────────┐
│                    第五层：交付与保障层                     │
│ 私有化 / 混合云 / 安全 / 审计 / 运维 / 监控 / 备份          │
└────────────────────────────────────────────────────────────┘
```

这 5 层里，真正最重要的是中间 3 层：

- 应用服务层决定“用户看到什么能力”
- 智能中台层决定“系统怎么把能力做出来”
- 业务与数据层决定“能力有没有真实业务价值”

## 5. 模块架构要具体到什么程度

如果要给领导看得更清楚，建议把整套系统拆成 6 个业务模块。

### 5.1 智能问答模块

这是用户最直观感受到的模块。

负责：

- 回答系统使用问题
- 回答技术知识问题
- 结合当前页面给出针对性说明
- 提供答案来源和引用依据

面向人群：

- 操作员
- 工程师
- 管理人员

### 5.2 报警与运维辅助模块

这是最容易体现业务价值的模块。

负责：

- 接收 DFMS 报警事件
- 识别报警对象、设备、产线、告警代码
- 自动匹配相关知识和历史案例
- 生成维修建议
- 通过 DFMS 或企业微信推送给相关人员

面向人群：

- 运维工程师
- 班组长
- 售后支持

### 5.3 自然语言数据分析模块

这是面向管理层和业务人员的重要模块。

负责：

- 自然语言查询生产指标
- 自动组织统计结果
- 输出图表和趋势分析
- 支持多轮追问

面向人群：

- 生产管理人员
- 运营管理人员
- 项目负责人

### 5.4 企业知识库模块

这是整套系统的知识底座。

负责：

- 接收和管理技术资料
- 把手册、规范、案例转化为可查询知识
- 支持知识审核和更新
- 保证知识可追溯、可维护

面向人群：

- 管理员
- 文档维护人员
- 运维专家

### 5.5 消息触达与协同模块

这是把能力真正送到用户手里的模块。

负责：

- 企业微信通知
- 工单提醒
- 报警推送
- 结果卡片触达
- 跳转回 DFMS 对应页面

面向人群：

- 一线工程师
- 管理层
- 值班人员

### 5.6 管理运营模块

这是保障系统长期可运营的模块。

负责：

- 用户权限配置
- 文档审核和发布
- 日志审计
- 使用统计
- 运行监控

面向人群：

- 系统管理员
- 项目运营人员
- 交付运维人员

## 6. 各模块之间是怎么协同的

为了避免看起来像“很多模块各干各的”，这里把协同关系写清楚。

```text
DFMS 页面 / 企业微信 / 管理后台
          │
          ▼
     应用服务入口
智能问答 / 报警辅助 / 数据分析
          │
          ▼
       智能中台
上下文识别 / 知识匹配 / 数据分析 / 消息触达
          │
          ▼
  DFMS 业务数据 + 企业知识库 + 用户权限
          │
          ▼
       返回结果给用户
答案 / 建议 / 图表 / 消息通知 / 工单跳转
```

这套协同逻辑意味着：

- 用户不直接面对复杂底层系统
- 业务系统和知识系统被统一组织
- 智能能力作为“中台”对外提供服务
- 所有结果最终回到 DFMS 或企业微信场景中

## 7. 三条最核心的业务闭环

这部分是领导最容易判断价值的地方。

### 7.1 闭环一：系统使用辅助闭环

```text
用户进入 DFMS 页面
   ↓
系统识别当前页面、模块和对象
   ↓
用户发起提问
   ↓
智能助手调用知识能力
   ↓
输出解释、操作建议和引用
```

用户看到的结果：

- 这个页面是做什么的
- 这个按钮怎么用
- 当前报警是什么意思
- 这个工单下一步怎么处理

管理价值：

- 培训成本下降
- 使用错误减少
- 系统上线接受度提升

### 7.2 闭环二：报警处理闭环

```text
DFMS 产生报警
   ↓
系统识别设备、报警代码、产线和责任人
   ↓
自动关联知识和维修步骤
   ↓
生成维修建议
   ↓
推送到 DFMS 页面或企业微信
   ↓
工程师处理并反馈
```

用户看到的结果：

- 可能原因
- 建议处理步骤
- 安全提示
- 参考文档来源

管理价值：

- 故障定位更快
- 停机时间更短
- 经验更容易沉淀

### 7.3 闭环三：经营分析闭环

```text
管理人员提出业务问题
   ↓
系统识别查询意图
   ↓
关联 DFMS 数据
   ↓
生成统计结果与图表
   ↓
支持继续追问
```

用户看到的结果：

- OEE 情况
- 停机原因分布
- 报警趋势
- 工单执行情况

管理价值：

- 查询效率提升
- 业务洞察更快获取
- 决策更容易基于数据展开

## 8. 企业微信在整体架构中的位置

企业微信不是这套系统的核心业务中台，但它是非常重要的“触达层”和“移动入口层”。

### 8.1 企业微信负责什么

- 负责消息触达
- 负责移动端进入
- 负责提醒和协同
- 负责把智能助手嵌入日常办公流程

### 8.2 企业微信不负责什么

- 不负责承载核心业务规则
- 不负责保存核心业务数据
- 不负责替代 DFMS 主系统
- 不负责替代统一权限体系

### 8.3 企业微信接入后的业务价值

最直接的价值体现在 4 个方面：

- 报警能第一时间触达到责任人
- 工单和结果通知更及时
- 智能助手可从移动端快速进入
- 管理人员可通过移动端快速查看重点信息

### 8.4 企业微信接入后的架构关系

```text
企业微信
工作台 / 通知消息 / 菜单入口
    │
    ▼
接入与触达层
身份识别 / 页面跳转 / 消息回推
    │
    ▼
DFMS 智能中台
问答 / 运维 / 分析 / 消息服务
    │
    ▼
DFMS 核心业务系统
报警 / 工单 / 设备 / 生产数据 / 知识库
```

## 9. 部署架构要怎么理解

管理层通常最关心两个问题：

1. 客户能不能本地部署
2. 会不会因为部署模式不同做成两套系统

答案是：

**可以支持两种部署模式，但保持一套业务架构。**

### 9.1 私有化部署架构

适合：

- 大客户
- 制造现场网络封闭客户
- 对数据安全要求高的客户

部署方式：

- DFMS 系统在客户本地
- 知识库在客户本地
- 智能中台在客户本地
- 企业微信作为访问与通知入口

管理意义：

- 安全边界清晰
- 更容易通过客户 IT 审核
- 更适合重点客户落地

### 9.2 混合云部署架构

适合：

- 中型客户
- 希望降低硬件投入的客户
- 希望更快上线的客户

部署方式：

- 业务数据仍在本地
- 核心业务系统仍在本地
- 部分智能生成能力可以调用云端

管理意义：

- 交付更快
- 成本更可控
- 容易形成标准化销售方案

### 9.3 两种模式的统一架构原则

- 入口一致
- 业务流程一致
- 知识库逻辑一致
- 权限体系一致
- 管理后台一致

差别只体现在“能力部署位置”，不体现在“产品逻辑”。

## 10. 领导层最关心的管理架构

这套系统如果要真正落地，必须具备清晰的管理能力。

### 10.1 用户与权限管理

必须明确：

- 谁能看什么知识
- 谁能查什么数据
- 谁能进入哪些功能
- 谁能管理和发布知识

### 10.2 知识运营管理

必须明确：

- 哪些文档可以入库
- 哪些内容需要审核
- 哪些版本在生效
- 哪些知识已经过时需要替换

### 10.3 运行与审计管理

必须明确：

- 谁在什么时候问了什么
- 系统给了什么回答
- 是否引用了依据
- 关键操作是否可追踪

### 10.4 运营效果管理

必须明确：

- 哪些问题最常被问
- 哪类知识最常被使用
- 哪些报警场景最有效
- 哪些模块使用率最高

这部分决定项目能不能从“能用”走向“可持续优化”。

## 11. 分阶段实施路径

为了控制范围并让领导更容易把握进度，建议按 4 个阶段实施。

### 阶段一：智能助手基础版

建设内容：

- DFMS 内嵌入口
- 基础知识问答
- 文档知识库
- 基础后台管理

可交付成果：

- 用户可以在 DFMS 页面中发起问答
- 企业文档可统一纳入知识库
- 管理员可上传、管理和维护文档

阶段意义：

- 先把“知识问答闭环”做通

### 阶段二：报警运维版

建设内容：

- 报警联动
- 维修建议
- 企业微信推送
- 工单辅助入口

可交付成果：

- 报警发生后可自动关联知识
- 工程师可在消息中收到建议
- 处理链路从“找资料”变成“直接拿建议”

阶段意义：

- 在最有价值的场景里体现效率收益

### 阶段三：经营分析版

建设内容：

- 自然语言查数
- 图表输出
- 多轮分析交互

可交付成果：

- 管理人员可直接提问生产数据
- 系统输出指标、图表和趋势分析

阶段意义：

- 从运维辅助延展到经营分析

### 阶段四：产品化推广版

建设内容：

- 标准交付方案
- 私有化 / 混合云模板
- 运维规范
- 商业化包装

可交付成果：

- 形成标准产品说明
- 形成标准部署方式
- 形成对外销售和推广材料

阶段意义：

- 从项目落地走向规模化复制

## 12. 这套架构对双方分别意味着什么

### 12.1 对伟立机器人

- DFMS 产品能力升级
- 提高产品智能化竞争力
- 增强客户粘性
- 增加新的销售增长点

### 12.2 对炽橙科技

- 获得一个真实制造业落地案例
- 形成行业可复制方案
- 从项目实施走向产品化输出

### 12.3 对双方合作

- 有机会形成联合解决方案
- 有机会沉淀标准架构和交付模板
- 有机会逐步扩展到更多工业场景

## 13. 建议的汇报口径

如果用于向领导汇报，可以把这套方案归纳成 4 句话：

1. 这不是一个单点 AI 功能，而是 DFMS 的智能化升级架构。
2. 这套架构同时服务系统使用、设备运维和数据分析三类核心场景。
3. 企业微信是移动入口和通知通道，DFMS 智能中台才是核心能力中心。
4. 项目可以先落地、再扩展、再产品化，不需要一开始做成大而全的平台。

---

## 14. 一句话结论

这套架构的本质，是围绕 DFMS 建设一个“统一入口、统一能力、统一数据、统一权限、可持续运营、可商业化复制”的企业级智能助手体系，而不是简单叠加一个问答功能。
