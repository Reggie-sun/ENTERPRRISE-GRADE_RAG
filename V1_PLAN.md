# V1 Plan

## 1. 项目目标

本项目的目标是先做出一个可运行的企业知识库问答 / RAG 系统 V1，用来验证从文档入库到问答返回的完整链路。

V1 目标聚焦在以下几个方面：

- 做一个企业知识库问答 / RAG 系统
- 支持上传 PDF / Word / Markdown 等文档
- 支持检索 + rerank + 问答
- 提供 API 接口，便于后续接前端或业务系统
- 后续再逐步扩展权限、OCR、评估、前端等能力

---

## 2. V1 必做范围

V1 只保留最核心、最小可跑通的链路，不做额外复杂设计。

必做范围如下：

- 文档上传
- 文档解析
- chunk 切分
- embeddings 生成
- 向量入库
- 检索 top-k
- rerank
- LLM 生成回答
- 返回引用片段

也就是说，V1 的交付重点不是“功能很多”，而是“整条链路能稳定跑通”。

---

## 3. V1 暂时不做

为了保证项目收敛，以下内容明确不进入 V1：

- 多租户权限
- 复杂工作流
- 多模态
- 语音输入
- 自动评估平台
- 高并发优化
- 精细化后台管理
- 分布式调度
- 复杂 Office 文档深度解析
- OCR 识别

这些内容可以作为 V2 / V3 的扩展项，V1 阶段先不引入。

---

## 4. 技术栈先定死

为了避免前期频繁切换技术路线，V1 阶段先固定以下技术选型：

- 后端：FastAPI
- 向量库：Qdrant
- Embedding：BGE-M3
- Reranker：bge-reranker-v2-m3
- LLM：先用 Ollama，后续再视情况切换到 vLLM
- 文档解析：先支持 PDF / Markdown / txt，复杂 Office 文档后补
- 部署方式：Docker Compose

V1 的原则是先用一套固定技术栈把系统跑通，再考虑替换和优化。

---

## 5. V1 成功标准

当以下目标全部满足时，可以认为 V1 成功：

- 能上传一个 PDF 文档
- 能完成文档解析、切分、向量化和入库
- 能基于用户问题完成检索、rerank 和回答生成
- 返回结果中能带引用片段
- 单机环境可稳定跑通

一句话总结：V1 成功的标准不是“做得很全”，而是“从上传文档到返回带引用答案的最小闭环已经完成”。
