# `/review`

用于代码审查和变更评审。

这份命令只保留 ECC `/code-review` 的核心思路：

- 先收集改动范围
- 先读完整文件和调用链
- 只报告高置信问题
- 问题优先于总结
- 重点看 bug、回归、契约风险和缺失测试

## 1. 审查流程

1. 看当前 diff。
2. 明确本次改动想解决什么。
3. 读完整相关文件，不要只盯 diff。
4. 读相邻调用方、schema、测试和文档。
5. 只输出高置信、会影响正确性或可维护性的发现。

## 2. 本仓库必查项

### 契约与兼容性

- 是否碰到了 `MAIN_CONTRACT_MATRIX.md` 的稳定字段或稳定语义
- 如果碰到了：
  - 文档是否同步
  - schema 是否同步
  - 主契约测试是否同步
  - 前端 `api/types.ts` 与 `api/client.ts` 是否同步

### Retrieval / RAG 一致性

- `retrieval / chat / SOP` 是否仍共用同一套 retrieval 主逻辑
- supplemental 判定是否被 endpoint 特化分叉
- chunk / rerank / router 变更是否有明确样本或 baseline 依据
- 是否把启发式字段误当成 ground truth

### 后端正确性

- FastAPI endpoint、schema、service 是否一致
- 受保护接口是否仍保留 auth / scope 限制
- 外部调用是否有超时、错误处理、降级逻辑
- 配置是否来自真源，而不是散落常量

### 前端正确性

- 页面展示是否与后端真实字段一致
- API 类型、client、页面调用是否同步
- 权限与路由守卫是否仍正确
- 是否引入重复 client 逻辑或临时字段映射

### 验证与测试

- 是否有针对性验证
- 是否缺少关键回归测试
- smoke / eval 是否应该补跑却没跑

### 改动范围控制

- 是否混入无关重构
- 是否在修单点问题时顺手改了无关模块
- 是否用新抽象掩盖本可局部修复的问题

## 3. 只报高信号问题

默认跳过这些噪音：

- 纯风格偏好
- 与本次改动无关的历史问题
- 低置信猜测
- “可以更优雅”但不影响正确性的建议

## 4. 输出格式

先列问题，再给总结。

```markdown
## 发现的问题

1. [阻塞] 问题标题
   - 位置：`path/to/file`
   - 原因：为什么这是实质问题
   - 影响：会带来什么行为回归 / 风险
   - 建议：最小修正方向

2. [重要] 问题标题
   - 位置：
   - 原因：
   - 影响：
   - 建议：

## 结论

- 结论：通过 / 需修改 / 阻塞
- 已核对：
- 残余风险：
- 测试缺口：
```

如果没有发现明确问题，要直接写：

```markdown
## 发现的问题

未发现明确问题。

## 结论

- 残余风险：
- 测试缺口：
```
